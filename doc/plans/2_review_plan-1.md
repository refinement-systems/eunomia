**Two things that will bite you**

1. **The context switch saves no FP/SIMD state, which forces a softfloat userspace — and your target base contradicts that.** The `TrapFrame` you describe is `{x[31], sp_el0, elr, spsr}` — general-purpose only, no `q0–q31`/`fpsr`/`fpcr`. Under preemptive multithreading (Phase 3), if any userspace thread touches a v register, a context switch silently corrupts it. And on a hardfloat AArch64 target, LLVM uses v registers even in "integer" code — `memcpy`/struct-move lowering via `ldp/stp` on q registers, autovectorized loops, `f64` locals. So a hardfloat userspace + GP-only `TrapFrame` = latent FP corruption that only surfaces under Phase 3 + the right codegen. You wrote "softfloat inherited," which is the correct mitigation — but you base the JSON off `aarch64-unknown-none` (line 366), and that base is *hardfloat* (neon + fp-armv8 enabled). The softfloat variant is `aarch64-unknown-none-softfloat` (`-neon,-fp-armv8`, softfloat ABI). So either base off `-none-softfloat` or set those features explicitly, and add a note that the GP-only `TrapFrame` *mandates* softfloat until the switch saves v-state. This is also a standing constraint worth recording given your graphics/GA/ML interests: hardware FP/NEON in userspace is off the table until the context switch (and the `TrapFrame` growth, far bigger than the tpidr bump) saves SIMD.

2. **Overriding `abort_internal()` will not make a panic reap as `STATUS_PANIC` under `panic=abort`.** Your terminus reasoning (lines 57–63, task 2.3) has a gap: under `panic=abort`, a normal single panic flows `panic!` → std's panic handler → runs the hook (prints) → `rust_panic` → `__rust_start_panic`, and `panic_abort`'s `__rust_start_panic` calls its own `abort()` (`core::intrinsics::abort()` → `udf`). It does **not** route through `crate::sys::abort_internal()`. So with only the override, a panic will `udf`-fault and reap as `faulted(...)`, not `exited(STATUS_PANIC)`. The override *does* catch the double-panic and `always_abort` paths (those go through `abort_internal`), just not the common one. The fix: since std owns the panic handler in a std binary, install a custom panic hook in runtime init (2.1) that calls the PAL `exit(STATUS_PANIC)` after emitting the message — the hook runs *before* `__rust_start_panic`, so it terminates cleanly with your sentinel. Keep the `abort_internal`/`exit` overrides too, for the explicit-abort and double-panic paths. Your 2.3 gate ("a panicking std binary reaps as STATUS_PANIC") is exactly the test that catches this, so it'll surface — but the stated mechanism won't pass it as written.

**Ordering / correctness**

3. **The heap lock belongs in 3.2, not 3.3.** The heap is `unsafe impl Sync` with no lock until 3.3, but 3.2 introduces `thread::spawn`/concurrent execution, and spawned threads allocate (the boxed closure, per-thread TLS blocks in 3.5 "over the verified heap," any user code). So between 3.2 and 3.3 you have concurrent allocation against a non-thread-safe allocator — a data race. The heap's lock doesn't need the futex stack; a raw spinlock (the same bootstrap-spinlock primitive 3.3 already needs) suffices. Move "lift the single-thread no-lock assumption" to 3.2, coupled with the first spawn.

4. **`x0` can't carry `Box<dyn FnOnce()>` — it's a fat pointer.** Line 293's parenthetical "(only one scalar arg `x0` — a boxed `FnOnce` ptr fits)" will truncate the vtable if taken literally. `Box<dyn FnOnce()>` is two words. You need the double-box (`Box<Box<dyn FnOnce()>>`) so the outer pointer is *thin* and fits in one scalar — which is exactly what the pthread/`motor` thread entry does, so mirroring `motor` gets it right, but the phrasing invites a one-word bug. Worth fixing in the text so the implementer boxes twice.

5. **Single-core + strict fixed-priority + no priority inheritance makes the userspace bucket spinlock a deadlock risk across priority levels.** Per rev2 §5.4 (strict 32-level preemptive, no donation), a higher-priority thread spinning on the futex bucket lock held by a *preempted lower-priority* thread will spin forever — the holder can't be scheduled while the spinner is runnable, and `yield` under strict priority won't drop to a lower level. The hold is tiny (check-and-enqueue), so the probability is low, but it's nonzero and it's a real wedge, not just inefficiency. For the MVP: make the bucket lock yield rather than pure-spin, and document that locks shared *across priority levels* aren't safe until the deferred kernel wait-set lands (which is the real fix, since waiting moves into the kernel). This is worth stating explicitly next to the §5.4 "servers run above clients" convention, because the futex emulation quietly assumes cooperating threads of comparable priority.

**Worth checking (smaller)**

6. **The growable-heap deferral reintroduces a bootstrap circularity.** The `.bss` arena neatly sidesteps it today: the allocator is up before you read the startup block, so `_start` can decode-then-build-argv/env. The moment you move to the deferred `heap` *grant*, you need to decode the grant table to find the heap untyped — but building argv/env `Vec`s needs the heap. The decoder is already no-alloc (borrows `buf@`), so the resolution is forced ordering: recv → decode (no-alloc, locate heap grant) → init allocator → *then* build argv/env Vecs. Worth pinning in the deferred-work item so it isn't rediscovered painfully.

7. **`restricted_std` polarity.** Double-check whether adding `eunomia` to the allowlist gives you full std vs. whether you want `restricted_std` + `#![feature(restricted_std)]` during bring-up. Both work; they differ in whether every downstream binary needs the feature attribute. Confirm against the 1.98 `build.rs` condition rather than the line-367 phrasing.

8. **`compiler-builtins-mem`.** 0.1 lists `-Zbuild-std=...,compiler_builtins,...` but not `-Zbuild-std-features=compiler-builtins-mem`. std's `fmt`/`io` paths emit more `memcpy`/`memset` than minimal `no_std`, so make sure the mem intrinsics are provided (either that feature or whatever the kernel/userspace already uses today — you clearly have *something* since `urt` links).

9. **"`MAX_ALIGN=64` → null = clean OOM" conflates two failures.** An `align > 64` request returning null isn't OOM, it's an unsupported-alignment ceiling reported as allocation failure (over-aligned types → abort even with free memory). Rare, and you disclosed the null behavior, but I'd label it an alignment ceiling rather than fold it into OOM so it isn't mistaken for a memory-pressure symptom later.

One optional efficiency note on 3.1: `TPIDR_EL0` only needs writing when *resuming a different thread*, since EL1 never clobbers it during a syscall — so strictly you only need it on the context-switch path, not every EL0 entry/exit. The uniform save/restore you describe is correct and safer, just slightly redundant; fine to keep for the MVP.
